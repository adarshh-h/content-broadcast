================================================
CONTENT BROADCASTING SYSTEM
Architecture Notes
================================================

1. AUTHENTICATION & RBAC FLOW
-------------------------------
JWT (JSON Web Token) based authentication is used.

Flow:
  POST /auth/login
  → verify email + password using bcrypt
  → generate JWT token (contains: userId, email, role)
  → client sends token in every request as:
     Authorization: Bearer <token>
  → authMiddleware verifies token
  → roleMiddleware checks role (principal/teacher)
  → controller handles request

Role Separation:
  - Principal → can only access /approval routes
  - Teacher   → can only access /content routes
  - Student   → hits public /content/live route (no auth)

Middleware Chain:
  Request → authMiddleware → roleMiddleware → Controller


2. SUBJECT-BASED SYSTEM DESIGN
--------------------------------
Each teacher can upload content for multiple subjects.
Each subject has its own independent rotation slot.

Tables involved:
  content_slots    → one slot per teacher per subject
                     UNIQUE(teacher_id, subject)
  content_schedule → rotation order and duration per content

Example:
  Teacher Rahul uploads:
  - Maths Paper 1, Maths Paper 2 → goes into Maths slot
  - Science Q1, Science Q2       → goes into Science slot

  content_slots:
  id | teacher_id | subject
  1  | 2          | maths
  2  | 2          | science

  content_schedule:
  id | content_id | slot_id | rotation_order | duration
  1  | 1          | 1       | 1              | 5
  2  | 2          | 1       | 2              | 5
  3  | 3          | 2       | 1              | 5
  4  | 4          | 2       | 2              | 5


3. UPLOAD HANDLING APPROACH
-----------------------------
- Multer middleware handles multipart/form-data uploads
- File validation: only JPG, PNG, GIF allowed (by mimetype)
- File size limit: 10MB enforced by multer
- Files stored locally in /uploads directory
- Unique filename: timestamp + random number
- File accessible via: GET /uploads/filename.jpg
- file_url stored in DB as /uploads/filename.jpg


4. APPROVAL WORKFLOW DESIGN
-----------------------------
Content Lifecycle:
  uploaded → pending → approved / rejected

Rules:
  - Teacher uploads → status automatically set to 'pending'
  - Only PRINCIPAL can approve or reject
  - Only PENDING content can be approved or rejected
  - Already approved content CANNOT be rejected
  - Already rejected content CANNOT be approved
  - Rejection MUST include a reason (validated)
  - approved_by and approved_at stored on action
  - Only 'approved' content is eligible for broadcasting
  - Even approved content must be within time window


5. SCHEDULING / ROTATION LOGIC (MOST IMPORTANT)
-------------------------------------------------
Algorithm:

  Step 1: Get all approved content for teacher
          (must have start_time and end_time)

  Step 2: Group content by subject
          grouped = {
            maths:   [Paper A, Paper B, Paper C],
            science: [Q1, Q2, Q3]
          }

  Step 3: For each subject independently:
    a. Filter: now >= start_time AND now <= end_time
    b. totalCycle = sum of all durations in milliseconds
    c. referenceStart = earliest start_time
    d. elapsed = now - referenceStart
    e. position = elapsed % totalCycle
    f. Loop through contents accumulating duration
       until position < cumulative → that is active content

  Step 4: Return 1 active content per subject

Example:
  Maths: A(5min), B(5min), C(5min) → total = 15min
  elapsed = 23min → 23%15 = 8min
  A covers 0-4, B covers 5-9, C covers 10-14
  8 falls in B range → show Paper B

  Science: X(3min), Y(3min) → total = 6min
  elapsed = 23min → 23%6 = 5min
  X covers 0-2, Y covers 3-5
  5 falls in Y range → show Y

  Maths and Science rotate COMPLETELY INDEPENDENTLY!


6. DATABASE DESIGN
-------------------
4 Tables:

  users
  → stores both principals and teachers
  → role field: 'principal' or 'teacher'

  content
  → main content table
  → status lifecycle: pending → approved/rejected
  → start_time, end_time for scheduling
  → uploaded_by → references users

  content_slots
  → groups content by subject per teacher
  → UNIQUE(teacher_id, subject) prevents duplicates
  → one slot per teacher per subject

  content_schedule
  → defines rotation order and duration
  → links content to slot
  → rotation_order determines sequence

Key Decisions:
  - UNIQUE constraint prevents duplicate slots
  - ON DELETE CASCADE cleans up related data
  - Indexes on status, subject, uploaded_by for performance
  - Timestamps stored as UTC in PostgreSQL

8. MIDDLEWARE USAGE
--------------------
cors()           → Allow cross-origin requests
express.json()   → Parse JSON request body
authMiddleware   → Verify JWT token
roleMiddleware   → Check principal/teacher role
multer           → Handle file uploads
express.static   → Serve uploaded files


9. SCALABILITY APPROACH
------------------------
Current (MVP):
  - Local file storage → can migrate to AWS S3
  - Stateless JWT → horizontally scalable
  - Connection pooling via pg Pool

Future improvements:
  - Redis caching for /content/live (reduce DB load)
  - Rate limiting on public API (prevent abuse)
  - AWS S3 for file storage (scalable + CDN)
  - Bull/Redis queue for approval notifications
  - Pagination for content listing APIs


10. ASSUMPTIONS
----------------
- start_time/end_time are full datetime values
- Content without start_time/end_time is never shown
- Default rotation duration is 5 minutes
- Invalid teacherId returns empty response (not 404)
- Subject names stored in lowercase
- Only PENDING content can be approved or rejected
